# Part 1: 从 GPT 的 Attention 说起 —— 问题在哪里

> **适合读者**：了解基础神经网络、知道 Transformer 是 GPT 的核心，但不熟悉 Attention 计算细节。
> **目标**：理解标准 Self-Attention 为什么慢，瓶颈在哪里，从而明白 FlashAttention 要解决什么。

---

## 1.1 GPT 的"注意力"是什么？

先一句话说清楚：**Attention 就是加权求和**。

GPT 读一段文本时，每个位置上的词都要"看"一遍前面的所有词，然后决定每个词对当前词的重要性（权重），最后按权重混合信息。

数学上，给定三个矩阵 Q、K、V（Query、Key、Value），Self-Attention 的计算是：

```
S = Q @ K^T      ← 计算相似度 (得分矩阵)
P = softmax(S)   ← 归一化为权重 (每行和为 1)
O = P @ V        ← 按权重混合信息
```

**形状**：假设序列长度 N = 4096，维度 d = 64，那么：

| 变量 | 形状 | 含义 |
|------|------|------|
| Q, K, V | (N, d) | 每个 token 的查询、键、值 |
| S | (N, N) | 4096×4096 的相似度矩阵 |
| P | (N, N) | softmax 后的权重 |
| O | (N, d) | 最终输出 |

**N 越大，问题越严重**。S 和 P 是 N×N 的，当 N = 4096 时还勉强，N = 128K 时，光存一个 S 就要 128K×128K×2≈32GB（bf16）—— 远超 GPU 显存。

---

## 1.2 复杂度：O(N²) 的噩梦

用大白话说复杂度：

1. **计算 S** = Q @ K^T：Q 的每一行 × K 的每一列 → N×N 次点积 → O(N²·d) 次运算
2. **计算 P** = softmax(S)：对 S 的每一行算 exp/求和 → O(N²) 次运算
3. **计算 O** = P @ V：P 的每一行 × V 的每一列 → O(N²·d) 次运算

所以总计算量 ≈ O(N²·d)。**当 N 翻倍，计算量翻 4 倍**。这就是 "Attention 是平方复杂度" 的含义。

但注意：**计算量不是唯一的瓶颈**。还有更隐蔽的——**内存访问（MAC）**。

---

## 1.3 GPU 的内存层次

GPU 和 CPU 一样有内存层次结构，速度越快的越小：

```
        🚀  SRAM  (计算单元旁边)    每个SM约192 KB  →  速度极快   (A100: ~19 TB/s 聚合带宽)
        ⬇       寄存器/Warp级缓存
        ⬇       共享内存 (Shared Memory)
        ⬇    
        💾  HBM  (显存主存)          80 GB          →  速度较慢   (A100: ~2 TB/s)
```

- **SRAM（共享内存）**：每个 SM 仅 ~192 KB，快到飞起，但极小
- **HBM（显存）**：几十 GB，但相对慢很多

**关键问题**：标准 Attention 在每一步之间都要把中间结果写回 HBM，再读出来：

```
Q, K (HBM) → GPU 计算 → S (写回 HBM)    ← 一次 HBM 写入！
S (HBM) → GPU softmax → P (写回 HBM)    ← 第二次 HBM 写入！
P, V (HBM) → GPU 计算 → O (写回 HBM)    ← 第三次 HBM 写入！
```

每一次 HBM 读写都是几百 GB/s 的带宽费用 —— 而 SRAM 的带宽是 HBM 的 ~10 倍。

---

## 1.4 看懂两种瓶颈（Roofline Model）

GPU 算子分为两类：

| 类型 | 特点 | 例子 | 速度瓶颈在哪儿 |
|------|------|------|---------------|
| **Compute-bound** | 计算量大，访存少 | 大矩阵乘法 (M>>d) | GPU 计算单元 |
| **Memory-bound** | 访存多，计算少 | softmax, dropout, 逐元素操作 | 显存带宽 |

```
计算密集型:  ████████████░░░░░░░░  80% 计算, 20% 访存
访存密集型:  ██░░░░░░░░░░░░░░░░  20% 计算, 80% 访存
```

看一个具体的数值例子。假设 N=4096, d=64：

| 操作 | FLOPs | 读写 HBM | 算术强度 (FLOPs/Byte) | 瓶颈 |
|------|-------|----------|----------------------|------|
| Q @ K^T | 2×4096×4096×64 ≈ 2.1G | ~2MB | ~1000 | **Compute** (快) |
| softmax | 3×4096×4096 ≈ 50M | ~134MB | ~0.37 | **Memory-bound** (慢！) |
| P @ V | 同 Q@K^T ≈ 2.1G | ~2MB | ~1000 | **Compute** (快) |

看到了吗？softmax 的 **计算量极低** 但 **访存量巨大**（因为 S 和 P 都是 N×N 矩阵，要读写 HBM）。FlashAttention 解决的就是这个 "中间结果放回 HBM" 的问题。

**更关键的是**：在 GPU 上 `Q@K^T` 和 `P@V` 是经过极致优化的 GEMM（通用矩阵乘法），利用率可以达到理论峰值的 80%+。但 softmax 的利用率不到 10%，因为它在"等数据"。

---

## 1.5 直觉类比

想象你要做一顿大餐：

```
标准 Attention:
  1. 去大冰箱 (HBM) 拿 Q, K → 切菜 (Q@K^T) → 放回大冰箱 (S 写回 HBM)
  2. 去大冰箱拿 S → 焯水 (softmax) → 放回大冰箱 (P 写回 HBM)
  3. 去大冰箱拿 P, V → 炒菜 (P@V) → 上桌 (O)

FlashAttention:
  1. 把所有菜一次性搬到操作台 (SRAM) 
  2. 在操作台上完成全部工序
  3. 最后只把成品放回大冰箱
```

**省掉的是来回跑大冰箱的时间**，而不是做菜本身。FlashAttention 甚至**多做了一些数学计算**（因为分块计算 softmax 需要额外修正），但由于省掉了巨量 HBM 访问，总体还是快得多。

---

## 1.6 关键总结

1. Standard Attention 的计算量是 O(N²·d)
2. 但更致命的瓶颈是 **HBM 读写**——中间结果 S(N×N) 和 P(N×N) 要反复在 HBM 和计算单元之间搬运
3. 大序列长度时，显存根本装不下中间的 N×N 矩阵
4. **FlashAttention 的核心思路**：不做全量计算，而是**分块（tiling）**，把小块数据搬到 SRAM 里一口气算完，减少 HBM 访问
5. 代价：需要**重新设计 softmax 算法**以支持分块计算 —— 这就是 Part 2 的内容
